# Prebuilt Docs Runtime

## Outcome

APCC now ships the docs site as a shared prebuilt viewer shell.

Each repository still has its own staged runtime data, but it no longer installs or builds a project-specific Next or Fumadocs application just to open the local site.

## Runtime Model

`apcc site open` now does four things:

1. stage the docs package into the runtime root
2. generate runtime data from staged docs and `.apcc`
3. start or reuse the shared prebuilt viewer shell
4. point that shell at the current runtime-data root

The shared shell is built once from `site-runtime/` and stored under:

`dist/site-runtime-prebuilt/`

The per-project runtime root only stores staged docs, runtime data, logs, and lifecycle metadata.

It should not carry its own `node_modules` or standalone server build artifacts.

Current runtime-state vocabulary:

- `live`: the shared shell is running for a project and can auto-refresh from runtime data
- `staged`: runtime data exists but no live site process is active
- `build`: a read-only deployable site artifact generated by `apcc site build`

## Runtime-Data Contract

The viewer shell renders from runtime data instead of from source-loader hooks.

Current contract:

- `runtime-data/control-plane.json`
- `runtime-data/docs-revisions.json`
- `runtime-data/docs-viewer.json`
- `runtime-data/version.json`

`docs-viewer.json` provides:

- navigation derived from staged docs
- per-page markdown payloads
- extracted headings
- page metadata used by the viewer shell and local search

The viewer shell now uses runtime data for:

- sidebar navigation
- authored page rendering
- TOC generation
- local search indexing
- docs revision previews
- console views derived from `.apcc`, including the plan-first execution tree with scoped task drilldown

## What Was Removed

The docs runtime no longer depends on:

- `getSource()`
- `.source/*`
- `source.config.ts`
- `fumadocs-mdx`
- per-project runtime `npm ci`
- per-project `next build`

The shared shell build is a maintainer/release concern. It runs through `npm run build` and is packaged under `dist/site-runtime-prebuilt/` before APCC is published. End users should not need to run a shell build before `apcc site open`.

`apcc site build` is not the shell build command. It copies the packaged shell and embeds a project snapshot into a deployable docs-site output directory.

## Sync Behavior

The docs site should keep the same editing feedback loop as before:

- editing authored docs updates `docs-viewer.json`
- revision state updates alongside authored docs changes
- `version.json` changes so the browser refresh layer can detect the update
- later authored docs edits must keep advancing `version.json`; the loop cannot break after the first automatic refresh
- repeated `apcc site open` calls reuse a healthy shell instead of restarting it

The watcher currently achieves this by polling the staged docs roots and restaging runtime data when the source workspace changes.

## Verification

Release-level verification for this runtime model is:

```bash
npm run check
npm run test
npm run build
npm run dev -- site build
npm run verify:site-lifecycle
```

The lifecycle smoke now verifies both:

- shared prebuilt shell reuse
- `site build` does not stop the live runtime
- authored docs sync after an on-disk edit
